# Unit Feedback Protocol

## Purpose

Evaluate feedback before acting on it. Not all suggestions improve a unit. Some conflict with
the benchmark, the framework, or your students' needs. Think before you revise.

**Rule: Understand before implementing. Verify before agreeing.**

## When to Use

- After an admin observation with notes on the unit
- After a peer or instructional coach reviews your materials
- After your own post-unit reflection
- Before any revision triggered by external feedback

## Step 1: Collect the Feedback

Ask: "Share the feedback you received. You can paste notes, describe what was said, or read it aloud."

Record each piece of feedback as a discrete item.

## Step 2: Evaluate Each Item

For each piece of feedback, run this evaluation:

**Benchmark alignment check:**
- Does this suggestion conflict with the Florida BEST benchmark requirements?
- Does it remove a required element (e.g., "remove the organizer" — organizer is core to scaffold)?
- If conflict: flag it. Do not implement blindly.

**Student needs check:**
- Does this suggestion account for your specific students (ESOL levels, reading levels, prior performance)?
- Is the suggestion based on observation of YOUR class or a generic best practice?

**Framework check:**
- Does this suggestion conflict with the 6-day IR cycle (bellringer → TL → IND structure)?
- Does it contradict the gradual release model (I Do → We Do → You Do)?
- If conflict: note the conflict, evaluate whether the framework exception is warranted.

## Step 3: Classify Each Item

| Classification | What it means | Action |
|----------------|--------------|--------|
| **Accept** | Technically sound, improves the unit | Add to revision list |
| **Modify** | Good idea but needs adjustment for your context | Note what adjustment is needed |
| **Reject** | Conflicts with benchmark/framework/student needs | Document reason, do not implement |
| **Clarify** | Unclear — need more information | Ask a follow-up question |

## Step 4: Report Back

Present your evaluation:

```
Feedback Evaluation: [Unit Name]

ACCEPT (will revise):
- "Add more sentence starters to the organizer" → Adding to You Do w/ Partner row

MODIFY (will revise with adjustment):
- "Make the bellringer shorter" → Will keep 3 words but reduce from 2 MC to 1 MC + 1 written

REJECT (will not implement):
- "Remove the I Do row — students need to figure it out themselves"
  Reason: I Do row is required scaffold per gradual release model. Removing it conflicts with
  ELA.10.R framework requirements.

CLARIFY (need more info):
- "The Day 3 activity feels too easy" → Which part? The organizer or the comprehension questions?
```

## Step 5: Route Accepted Items to Revision

Pass accepted + modified items to unit-reviser with the evaluation as context.
